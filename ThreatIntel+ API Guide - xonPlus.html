Partner API

Overview

The ThreatIntel+ Partner API provides programmatic access to all threat intelligence capabilities for partners, MSSPs, and service providers. This comprehensive RESTful API enables you to integrate threat intelligence data into your existing security tools, automate client onboarding workflows, and build custom dashboards tailored to your operational needs.

The Partner API is designed for high-volume operations with support for bulk queries, batch processing, and efficient data retrieval across multiple client accounts. Whether you're integrating with a SIEM platform, building custom reporting tools, or automating security workflows, the Partner API provides the flexibility and power you need.

API Key Management

The Partner API interface provides comprehensive tools for creating, monitoring, and managing your API keys. Proper API key management is critical for secure access to threat intelligence data and maintaining operational security.

Key Generation

Create New API Key

Generate new API keys with a single click. Each key is uniquely generated and displayed with creation timestamp, creator information, and initial active status. Keys are immediately ready for use in API requests.

Key Metadata

Each API key includes comprehensive metadata for tracking and management:

  • Unique key identifier for API authentication
  • Creation timestamp for audit logging
  • Creator information (user who generated the key)
  • Current status (Active or Suspended)
  • Usage statistics and request counts

Key Suspension

Suspend Active Keys

Temporarily disable API keys without deleting them. Suspended keys cannot be used for API requests but retain all historical data and can be reactivated at any time. Use this feature for temporary access control or when investigating potential security issues.

Suspension Status

Suspended keys are clearly marked with a red "Suspended" badge and X icon in the management table. Any API requests using suspended keys will be rejected with an appropriate error message.

Key Activation

Reactivate Keys

Restore functionality to suspended API keys with the activate button. Reactivated keys immediately regain full API access capabilities and can be used for requests. All historical usage data is preserved during suspension and reactivation.

Active Status

Active keys display a green "Active" badge with checkmark icon, indicating they are ready for use in API requests. Monitor active keys regularly for unusual usage patterns or unauthorized access.

API Usage Monitoring

Track and analyze your Partner API usage with comprehensive monitoring tools and metrics. The API interface provides real-time visibility into request volumes, quota usage, and historical trends across all your API keys.

Usage Metrics

Current Usage

Real-time display of API request consumption showing current usage versus total allocated quota. Visual progress bar with color coding indicates usage levels: green (normal), yellow (moderate), red (approaching limit).

Historical Trends

Interactive charts showing API usage patterns over the last 30 days. Identify peak usage periods, track growth trends, and optimize your API consumption strategy based on historical data.

Per-Key Statistics

Detailed usage statistics for each API key, helping identify which keys are most active and detect any unusual patterns or potential security issues.

Domain Management Endpoints

The Domain Management endpoints allow you to programmatically onboard and manage domains for threat intelligence monitoring. You can add domains individually or in bulk, and track their validation status through the API. All endpoints require authentication via the X-API-Key header.

Single Domain Upload

Submit a single domain for onboarding and validation. This endpoint is ideal for adding individual domains on-demand or integrating domain onboarding into automated workflows. The domain will be queued for validation and processing.

Endpoint:

POST /v2/partner/domains

Headers:

X-API-Key: YOUR_API_KEY
Content-Type: application/json

Request Body:

{
  "domain": "example.com"
}

Request Example:

curl -X POST "https://plus-api.xposedornot.com/v2/partner/domains" \
  -H "X-API-Key: abc123...xyz789" \
  -H "Content-Type: application/json" \
  -d '{"domain": "example.com"}'

Success Response (202 Accepted):

{
  "message": "Domain accepted for validation",
  "domain": "example.com",
  "status": "pending_validation"
}

Bulk Domain Upload

Upload multiple domains at once using a file. This endpoint accepts CSV or TXT files containing one domain per line. Ideal for initial onboarding of multiple client domains or batch operations. The API validates each domain and provides detailed feedback on accepted, rejected, and duplicate entries.

Endpoint:

POST /v2/partner/upload-domains

Headers:

X-API-Key: YOUR_API_KEY
Content-Type: multipart/form-data

Request Example:

curl -X POST "https://plus-api.xposedornot.com/v2/partner/upload-domains" \
  -H "X-API-Key: abc123...xyz789" \
  -F "file=@/path/to/domains.txt"

Success Response (202 Accepted):

{
  "message": "Domains accepted for validation. 1 domains queued for processing.",
  "summary": {
    "total_submitted": 1,
    "plan_limit": 50,
    "current_quota_used": 1,
    "available_slots": 49,
    "accepted": 1,
    "rejected_invalid": 0,
    "rejected_denied": 0,
    "rejected_duplicate": 0,
    "rejected_quota": 0
  },
  "rejected_details": {},
  "duplicate_domains": [],
  "quota_rejected_domains": []
}

Response Fields:

  • total_submitted: Number of domains in the uploaded file
  • accepted: Number of domains successfully queued for validation
  • rejected_invalid: Domains with invalid format
  • rejected_duplicate: Domains already in your account
  • rejected_quota: Domains rejected due to quota limits

List All Domains

Retrieve a complete list of all domains associated with your partner account, including their current validation status, processing state, and metadata. Use this endpoint to monitor the status of onboarded domains and track your quota usage. Supports filtering and pagination.

Endpoint:

GET /v2/partner/domains

Headers:

X-API-Key: YOUR_API_KEY

Query Parameters:

  • domain (optional): Filter by domain name. Supports exact match or wildcard matching with * (e.g., *example*, example*, *.com)
  • domain_status (optional): Filter by validation status: pending_validation, approved, rejected, pending_approval
  • processing_status (optional): Filter by processing status: pending, processing, completed, failed
  • limit (optional, integer): Number of results to return. Range: 1-500. Default: 100
  • offset (optional, integer): Number of results to skip for pagination. Default: 0

Request Example:

curl -X GET "https://plus-api.xposedornot.com/v2/partner/domains" \
  -H "X-API-Key: abc123...xyz789"

Request Example (with filters):

curl -X GET "https://plus-api.xposedornot.com/v2/partner/domains?domain_status=approved&limit=10&offset=0" \
  -H "X-API-Key: abc123...xyz789"

Success Response (200 OK):

    "tier": "Ultimate",
    "domain_limit": 50,
    "current_quota_used": 2,
    "available_slots": 48
  },
  "total_count": 2,
  "domains": [
    {
      "entry_id": "5111641913950208",
      "domain": "example.com",
      "status": "approved",
      "processing_status": "completed",
      "submitted_at": "2025-12-07T12:05:56.522893+00:00",
      "verified_at": "2025-12-07T12:05:56.707645+00:00",
      "rejection_reason": null,
      "source": "csv",
      "record_count": 1
    },
    {
      "entry_id": "5640572144451584",
      "domain": "example.org",
      "status": "approved",
      "processing_status": "completed",
      "submitted_at": "2025-12-13T09:26:08.690515+00:00",

      "source": "api_bulk",
      "record_count": 727
    }

}

Domain Status Values:

  • pending_validation: Domain is queued for validation
  • pending_approval: Domain requires admin approval (large domains)
  • approved: Domain has been validated and approved
  • rejected: Domain was rejected (see rejection_reason)

Processing Status Values:

  • pending: Processing has not started yet
  • processing: Domain data is currently being processed
  • completed: Processing is complete, data is available
  • failed: Processing failed (contact support)

Source Values:

  • api: Added via single domain API endpoint
  • api_bulk: Added via bulk upload API endpoint
  • csv: Added via dashboard file upload
  • manual: Added manually via dashboard

Threat Intelligence Endpoints

The Threat Intelligence endpoints provide access to breach data and exposure information for emails and domains. Use these endpoints to check for compromised credentials, retrieve breach summaries, and generate security reports. All endpoints require authentication via the X-API-Key header.

Email Breach Check

Query the breach database to check if an email address has been exposed in any known data breaches. By default, returns a list of breach IDs. Use the ?detailed=true query parameter to get full breach information including exposed data types, breach dates, and risk assessments.

Endpoint:

GET /v3/check-email/{email}

Query Parameters:

  • detailed (optional, boolean): Set to true to include full breach details. Defaults to false.

Request Example (Simple):

curl -X GET "https://plus-api.xposedornot.com/v3/check-email/user@example.com" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json"

Simple Response (default):

Returns only breach IDs as a list of strings:

{
  "status": "success",
  "email": "user@example.com",
  "breaches": [
    "ExampleBreach",
    "AnotherBreach"
  ]
}

Request Example (Detailed):

curl -X GET "https://plus-api.xposedornot.com/v3/check-email/user@example.com?detailed=true" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json"

Detailed Response (with ?detailed=true):

{
  "status": "success",
  "email": "user@example.com",
  "breaches": [
    {
      "breach_id": "ExampleBreach",
      "breached_date": "2023-01-15T00:00:00+00:00",
      "logo": "https://xposedornot.com/static/logos/ExampleBreach.png",
      "password_risk": "hardtocrack",
      "searchable": "Yes",
      "xposed_data": "Email addresses;Passwords;Names",
      "xposed_records": 5000000,
      "xposure_desc": "Description of the breach incident",
      "domain": "example.com"
    }
  ]
}

Domain Breach Summary

Retrieve aggregated breach data for an entire domain, including yearly metrics, top breaches, and breach timeline. Ideal for client reporting and executive dashboards. Use ?details=true to include per-breach details.

Endpoint (Summary):

GET /v2/partner/domain-summary?domain={domain}

Headers:

X-API-Key: YOUR_API_KEY

Query Parameters:

  • domain (required): Domain name to check
  • details (optional, boolean): Set to true to include detailed breach information. Defaults to false.

Request Example:

curl -X GET "https://plus-api.xposedornot.com/v2/partner/domain-summary?domain=example.com" \
  -H "X-API-Key: YOUR_API_KEY"

Endpoint (Detailed):

GET /v2/partner/domain-summary?domain={domain}&details=true

Request Example:

curl -X GET "https://plus-api.xposedornot.com/v2/partner/domain-summary?domain=example.com&details=true" \
  -H "X-API-Key: YOUR_API_KEY"

      }

  "Domain_Summary": {
    "example.com": 1250
  },
  "Breach_Summary": {
    "ExampleBreach": 500,
    "AnotherBreach": 750
  },
  "Top10_Breaches": {
    "AnotherBreach": 750,
    "ExampleBreach": 500
  },
  "Detailed_Breach_Info": {
    "ExampleBreach": {
      "breached_date": "2023-01-15",
      "xposed_data": "Email addresses;Passwords",

"Breaches_Details": [

Response Fields:

  • breach_count: Total number of breaches across all years

  • Domain_Summary: Exposed email count per domain

When the domain is currently being processed or not yet complete:

{

When the domain has not been approved for this partner:

{
  "detail": "Domain not approved for this partner"
}

Authentication

API Key Authentication

All Partner API requests require authentication using an API key. Include your API key in the x-api-key header of every request.

Header Format:

x-api-key: YOUR_API_KEY_HERE

Example with curl:

curl -X GET "https://plus-api.xposedornot.com/v3/check-email/user@example.com" \
  -H "x-api-key: abc123def456ghi789jkl" \
  -H "Content-Type: application/json"

Important Security Notes:

  • Never expose API keys in client-side code or public repositories
  • Rotate API keys regularly as part of security best practices
  • Use separate API keys for different environments (dev, staging, production)
  • Suspend keys immediately if you suspect unauthorized access
  • Monitor key usage for unusual patterns or unexpected activity

Error Handling

Error Responses

The Partner API uses standard HTTP status codes and returns detailed error messages in JSON format to help you diagnose and resolve issues quickly.

400 Bad Request

Invalid request format or parameters

{
  "detail": {
    "status": "error",
    "message": "Invalid email format"
  }
}

401 Unauthorized

Invalid or missing API key

{
  "detail": {
    "status": "error",
    "message": "Invalid API key"
  }
}

403 Forbidden

Suspended API key or insufficient permissions

{
  "detail": {
    "status": "error",
    "message": "API key suspended"
  }
}

429 Too Many Requests

Rate limit exceeded

{
  "detail": {
    "status": "error",
    "message": "Rate limit exceeded. Please try again later."
  }
}

Rate Limits & Quotas

Request Limits

Partner API access is subject to rate limits and monthly quotas based on your subscription tier. Monitor your usage through the API interface to avoid hitting limits.

  • Rate Limit: Maximum requests per minute varies by tier
  • Monthly Quota: Total API calls per month based on subscription
  • Burst Allowance: Short-term burst capacity for peak loads
  • Quota Reset: Monthly quota resets on subscription renewal date

Best Practices:

  • Implement exponential backoff for retries on rate limit errors
  • Cache frequently accessed data to reduce API calls
  • Use bulk endpoints when checking multiple items
  • Monitor usage regularly and plan for quota increases if needed

Best Practices

Security

  • Never hardcode API keys in source code or configuration files
  • Use environment variables or secure key management systems
  • Rotate API keys regularly (every 90 days recommended)
  • Create separate keys for different environments and applications
  • Suspend unused keys immediately to reduce security exposure
  • Monitor key usage for anomalies that could indicate compromise

Development

  • Implement proper error handling for all API responses
  • Use bulk endpoints for batch operations to optimize quota usage
  • Cache results appropriately to minimize redundant API calls
  • Implement exponential backoff for rate limit retries
  • Log API requests and responses for debugging and auditing
  • Test thoroughly in development before deploying to production
